01 · The Product

What TechClear is.

A marketing landing site + lead-capture + enrollment platform for the PM + AI Flagship Program — an 8-week IT workforce training course that combines SAFe Scrum Master 6.0 certification with hands-on AI tooling.

The Course

8 weeks

Cohort-based, SAFe SSM 6.0 certification included, in-person kickoff in Tysons, VA.

Price

$3,000

One-time payment. Promotion codes enabled. No installments.

Launch

July 6, 2026

First cohort kickoff date TBA. Announced site-wide in the top banner + hero block.

One-line summary: A single-page Next.js site that captures inquiries into HubSpot, takes payments via Stripe (and Whop in future), and sends confirmation emails via Resend. Everything funnels into HubSpot as the system of record.

05 · Design System

The visual language.

Sharp, black, editorial. Intentional design choices that show up everywhere in the codebase — match them when you add new sections.

Typography

  • Playfair Display — all headings, italic accents for emphasis
  • Inter — all body copy, labels, buttons
  • Labels use tracking-[0.2em] uppercase text-[10px] in bordered pills

Color

  • Background: pure black #000
  • Section alternates: #050505, #020202
  • Text: #f4f4f5 primary, zinc-300/400/500 secondary
  • No brand color — white is the accent

Shape

  • rounded-none everywhere — no border radius, intentionally sharp
  • Three button levels: solid white, outlined white, dim outline
  • Cards: hairline borders border-white/15 on near-black backgrounds

Motion

  • Framer Motion for everything animated
  • mounted state guard prevents hydration double-fire on mobile
  • whileInView with amount: 0.2 threshold for scroll triggers
  • Easing: [0.16, 1, 0.3, 1] cubic bezier
Custom cursor: Desktop-only animated cursor (glow + ring + dot) lives in components/CustomCursor.tsx. Disabled on /register via a .no-custom-cursor class. Hidden automatically on touch devices.

Three live services. One in transition.

07 · HubSpot

HubSpot is the system of record.

Every form submission and every payment lands here. If something writes to HubSpot and that write fails, the user gets an error. Everything else (emails, notes) is best-effort.

Functions — lib/hubspot.ts

  • createOrUpdateContact()
    Idempotent: tries create, on 409 conflict searches by email + updates. Sets email, firstname, lastname, phone, lifecyclestage, hs_lead_status.
  • createContactNote()
    Creates a HubSpot note object, attaches it to the contact via association type 202.
  • addContactToList()
    Defined but currently unused.

Lifecycle State

TriggerStageStatus
Form submitleadNEW
Stripe paymentcustomerCONNECTED

Lead → customer transition happens automatically via the Stripe webhook when payment succeeds.

Env var: HUBSPOT_ACCESS_TOKEN — Personal Access Token from HubSpot dashboard. Required scope: crm.objects.contacts.read/write, crm.objects.notes.read/write.
08 · Stripe

Stripe — Payment Link approach.

We don't create checkout sessions via API. We use a hosted Payment Link, redirect users to it, and listen for the resulting webhook. Simpler, no session lifecycle to manage.

Payment Link

https://buy.stripe.com/6oU14mgs2bIogALaim7wA01

Hardcoded in app/register/page.tsx. Promotion codes enabled in the Stripe Dashboard on the Link itself.

Webhook — what gets received

  • Event: checkout.session.completed
  • Signature verified with STRIPE_WEBHOOK_SECRET
  • Session retrieved with expand: ['line_items']
  • Extracted: email, fullName, phone, amount_total, line item description

What happens after

  • HubSpot contact upgraded to customer / CONNECTED
  • HubSpot note: "Payment received — [product] · $3,000.00 · Phone: [...]"
  • Receipt email sent to customer
  • Internal notification email to team
  • Always returns 200 to Stripe — email failures don't fail the webhook
Dashboard config required: In Stripe Dashboard → Developers → Webhooks, register endpoint https://techclear.io/api/webhooks/stripe for the checkout.session.completed event. Copy the signing secret into STRIPE_WEBHOOK_SECRET.

CHAPTER THREE

What happens when a user...

10 · Flow A

...submits the contact form.

STEP 01 · CLIENT
User fills form (firstName + email required, rest optional) → handleWaitlistSubmit sends POST to /api/waitlist
app/page.tsx
STEP 02 · API VALIDATION
Server checks firstName + email present, otherwise returns 400
app/api/waitlist/route.ts
STEP 03 · HUBSPOT WRITE
createOrUpdateContact with lifecyclestage='lead', leadStatus='NEW'. If contact exists (409), searches by email and updates instead.
lib/hubspot.ts
STEP 04 · HUBSPOT NOTE
If interest or message present, attaches a note: "Area of Interest: [label]\n\nMessage: [body]"
lib/hubspot.ts
STEP 05 · EMAIL (NON-BLOCKING)
Promise.allSettled fires two emails in parallel: user confirmation + internal notification. Failures here don't break the response.
lib/resend.ts + lib/mailer.ts
STEP 06 · CLIENT FEEDBACK
200 → formState = 'success' → renders "Thank you, [firstName]." card with reset button. 4xx/5xx → 'error' state with message.
app/page.tsx
11 · Flow B

...completes payment.

STEP 01 · CLIENT
User clicks "Enroll" → navigates to /register → that page immediately redirects to the Stripe Payment Link.
app/register/page.tsx
STEP 02 · STRIPE HOSTED CHECKOUT
User enters email, name, phone, card. Stripe handles 3DS, card validation, fraud checks, promo codes.
stripe.com (hosted)
STEP 03 · WEBHOOK FIRES
Stripe POSTs checkout.session.completed to /api/webhooks/stripe with signature header.
stripe → /api/webhooks/stripe
STEP 04 · VERIFY + EXPAND
Signature verified with STRIPE_WEBHOOK_SECRET. Session retrieved with expand:['line_items'] to get product details.
app/api/webhooks/stripe/route.ts
STEP 05 · HUBSPOT UPGRADE
createOrUpdateContact upgrades lifecyclestage='customer', leadStatus='CONNECTED'. Note attached with payment details.
lib/hubspot.ts
STEP 06 · EMAILS
Receipt to customer + internal notification to team. Same allSettled pattern — non-blocking.
lib/resend.ts
STEP 07 · RESPONSE
Returns 200 to Stripe regardless of downstream errors (Stripe retries on non-200, so we always ack).
app/api/webhooks/stripe/route.ts

17 · Risks

What I'd worry about.

SECURITY · Secrets in repo. Verify .env.local never made it into git history. If it did: rotate every key on HubSpot, Stripe, and Resend dashboards before doing anything else.
RELIABILITY · Email failures are silent. Both flows use Promise.allSettled on the email calls — the user sees success even if Resend is down. If a customer pays $3,000 and never gets a receipt, that's a support ticket waiting to happen. Consider piping email failures to Sentry or similar.
RELIABILITY · No idempotency on Stripe webhook. If Stripe retries a webhook (which it will on any non-200 response, network blip, or timeout), the contact gets a duplicate note and the customer may get a duplicate receipt email. The HubSpot upsert handles the contact correctly, but notes and emails are not idempotent.
UX · Single-page architecture. Everything lives in one app/page.tsx file. ~1000 lines. Easy to break with a stray edit, hard to code-review in chunks. Worth extracting sections into components/sections/ if you plan substantial changes.
STRATEGIC · Two payment paths. Stripe and Whop both wired (one half-wired). Decide before launch which is canonical — running both creates reconciliation headaches in HubSpot (Stripe webhook upgrades to customer, Whop won't unless you add API integration).

Your first week.

18 · Onboarding

Day-by-day kickoff.

If you've never seen this codebase before, this is the order I'd work through it.

Day 1 — Get it running

  • Clone repo, npm install
  • Get .env.local from a teammate (don't commit it)
  • npm run dev, load localhost:3000, click everything
  • Submit the contact form with your own email — verify HubSpot contact appears + you get the confirmation email

Day 2 — Read the code

  • Read CLAUDE.md end-to-end
  • Skim app/page.tsx top to bottom — get a feel for the sections
  • Read each lib/*.ts file
  • Trace one form submission from button click → HubSpot record

Day 3 — Audit dashboards

  • Log into HubSpot, Stripe, Resend, Whop — get accounts/permissions
  • Verify Stripe webhook is registered + delivering
  • Verify Resend domain status
  • Check HubSpot for the last 10 contacts — does the data look right?

Day 4 — Address the punch list

  • Create the OG image
  • Move the hardcoded email to an env var
  • Verify .env.local isn't in git history
  • Pick a stance: Stripe or Whop, and align all Enroll buttons

Day 5 — Document & commit

  • Update CLAUDE.md with anything you changed or learned
  • Create your first PR — even a small one — to confirm CI/deploy flow works
  • Set up Sentry or your error-monitoring tool of choice

Ongoing — Keep this true

  • CLAUDE.md is the living source of truth — update it when you change anything material
  • Test the contact form + Stripe checkout (test mode) before every deploy
  • Watch HubSpot lifecycle transitions in the first week of cohort to confirm flows still work end-to-end
